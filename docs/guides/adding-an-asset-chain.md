# Adding an Asset Chain

This guide describes how to add a reviewed asset catalog for another EVM
chain. A chain catalog provides convenient identities such as
`arbitrum.USDC`; it does not declare that a token is supported by Aave or safe
for a particular transaction.

## Architecture Boundary

The catalog pipeline separates reusable behavior from chain-owned inputs:

| Layer | Responsibility | Reuse for a new chain |
| --- | --- | --- |
| `config.Chain` | SDK chain identity and chain ID | Add one entry |
| source extractor | Read one pinned upstream export | Add or configure per chain |
| source adapter | Convert provider data to neutral candidates | Reuse for the same provider |
| `internal/assetmanifest` | Validate and evolve reviewed manifests | Reuse unchanged |
| `assets.Catalog` | Immutable lookup and ordering | Reuse unchanged |
| `internal/catalogloader` | Convert a manifest to `token.Ref` values | Reuse unchanged |
| `assets/<chain>` | Embed one manifest and expose chain API | Add a thin package |
| `internal/catalogcodegen` | Generate named Go references | Reuse unchanged |

The current neutral layers are multi-chain. The update command, Base
extractor, Makefile targets, integration environment, and scheduled workflow
still orchestrate only Base. Adding the second chain therefore includes
generalizing those entrypoints; copying Base-specific orchestration under a
new name is not the intended final design.

## 1. Register the SDK Chain

Add the chain to `config/chain.go` and append it after existing constants so
existing `config.Chain` numeric values do not change:

```go
const (
	Base Chain = iota
	Arbitrum
)

var ChainInfo = map[Chain]struct {
	Name    string
	ChainID int
}{
	Base:     {"Base", 8453},
	Arbitrum: {"Arbitrum", 42161},
}
```

Add the gas-token decimals and tests for `Name`, `ChainID`,
`GasTokenDecimals`, and `ChainIDToChain`. Register protocol deployments or
executor contracts only when the corresponding SDK feature supports the new
chain. Asset-catalog membership alone must not imply protocol support.

## 2. Define a Trusted Upstream Export

Prefer a versioned, immutable upstream artifact. For an Aave Address Book
market, define the expected export name and chain ID:

```go
definition := aaveaddressbook.ExportDefinition{
	Name:    "AaveV3Arbitrum",
	ChainID: 42161,
}
```

The extractor must:

- load the exact pinned `@aave-dao/aave-address-book` package;
- verify package metadata and the pinned Git commit;
- import the expected market export explicitly;
- verify its chain ID;
- emit underlying token addresses, not aToken or debt-token addresses;
- reject malformed or zero addresses;
- attach issuer provenance only when it comes from a reviewed HTTPS source.

Do not fetch a mutable branch during SDK runtime. JavaScript and network access
belong only to the update workflow; normal Go builds use checked-in artifacts.

For another data provider, add a separate source adapter. Do not add
provider-specific fields or validation to `internal/assetmanifest`.

## 3. Extend Update Orchestration

Before adding the second chain, change `cmd/update-aave-manifest` to iterate
over explicit chain specifications. Each specification should own at least:

```text
SDK chain
chain ID
upstream export definition
extractor command
manifest output path
generated Go output path
generated Go package name
```

For each chain, the command must perform this sequence:

```text
extract pinned upstream data
  -> source adapter
  -> assetmanifest.Generate
  -> assetmanifest.ValidateEvolution
  -> catalogcodegen.Generate
  -> atomic writes of manifest.json and catalog_gen.go
```

Keep `ValidateEvolution` enabled. Routine automation may add an asset, but it
must fail if an existing public ID is removed, assigned a different upstream
key, or retargeted to a different address. Those changes require an explicit
migration and deprecation decision.

Run the update command twice and verify that the second run produces no diff.

## 4. Add the Thin Chain Package

Create:

```text
assets/arbitrum/
  catalog.go
  catalog_gen.go
  manifest.json
```

`catalog.go` is handwritten. It may only:

- embed `manifest.json`;
- supply the expected chain and source definition;
- call `catalogloader.Load`;
- delegate `Lookup` and `Entries` to the shared catalog;
- provide `mustLookup` for generated declarations.

It must not contain a handwritten asset list or duplicate manifest validation,
sorting, lookup, or evolution logic.

`catalog_gen.go` is generated from the reviewed manifest:

```go
// Code generated by update-aave-manifest; DO NOT EDIT.

package arbitrum

var (
	USDC = mustLookup("USDC")
	WETH = mustLookup("WETH")
)
```

Never edit this file manually. An asset becomes a named public reference only
through a reviewed manifest change and regeneration.

## 5. Preserve Catalog Semantics

Catalog IDs are stable SDK keys. They are exact and case-sensitive; they are
not on-chain symbols. The executable identity is always chain plus contract
address.

The manifest may contain:

- the SDK ID;
- the upstream key;
- the underlying token address;
- optional reviewed issuer provenance;
- immutable upstream package and commit provenance.

The manifest must not contain:

- symbol, name, or decimals fetched from the token contract;
- aToken, debt-token, oracle, or other protocol-role addresses;
- reserve active, frozen, paused, or borrowing status;
- claims that the asset is safe or supported for execution.

Those values belong to runtime metadata or a protocol-owned registry. Unknown
tokens remain usable through `token.NewRef`; the reviewed catalog is a
convenience API, not an allowlist.

## 6. Add Tests

Add deterministic unit coverage for:

- source export parsing and chain-ID validation;
- byte-for-byte manifest generation;
- safe and unsafe manifest evolution;
- generated-source consistency;
- exact ID lookup, deterministic ordering, immutable `Entries`, and chain
  scoping.

The chain package test must regenerate source in memory and compare it with
`catalog_gen.go`. This keeps `go test ./...` sufficient to detect stale or
hand-edited generated code without requiring npm.

Add an integration test against a fork of the target chain. Resolve at least
two generated references through the public protocol registry and assert:

- the returned underlying `token.Ref` equals the generated reference;
- the chain and address are correct;
- representative runtime metadata is loaded from the chain, not the manifest.

Add chain-specific Anvil and integration targets rather than overloading
`BASE_RPC_URL`. For example, use `ARBITRUM_RPC_URL`, `anvil-arbitrum`, and a
documented hardfork compatible with the features exercised by the test.

## 7. Extend Automation

The scheduled update workflow must include every reviewed output for the new
chain in its pull-request diff:

```text
assets/arbitrum/manifest.json
assets/arbitrum/catalog_gen.go
```

It must run unit tests and the target-chain integration test before opening a
draft pull request. Generated public API changes must remain visible to human
reviewers, and the workflow must never auto-merge catalog changes.

Use a chain-specific RPC secret. Fail clearly when it is missing; do not skip
on-chain validation silently.

## Review Checklist

- The chain constant was appended without changing existing numeric values.
- The upstream package version and Git commit are pinned.
- The extractor validates the expected export and chain ID.
- Only underlying token identities enter the manifest.
- Neutral catalog and manifest packages remain provider- and chain-agnostic.
- `catalog.go` contains no asset declarations.
- `catalog_gen.go` exactly matches the reviewed manifest.
- Existing IDs were not removed, renamed, or retargeted implicitly.
- Unit, generated-consistency, integration compile, and target-chain fork tests
  passed.
- The automation opens a draft PR with manifest and generated API diffs.
